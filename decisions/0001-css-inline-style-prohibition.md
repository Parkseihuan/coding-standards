## 컨텍스트

16개 GitHub 프로젝트 분석 결과, 4개 프로젝트(retirement-system, Severance-pay, work-planner, yiu-attendance)에서 인라인 스타일을 사용하고 있었습니다.

**문제점:**
- 스타일이 HTML에 흩어져 있어 일관성 유지 어려움
- 디자인 변경 시 모든 HTML 파일 수정 필요
- CSS 변수 사용 불가능
- 다크 모드 등 테마 전환 구현 어려움
- 코드 재사용성 저하

## 결정

**모든 프로젝트에서 인라인 스타일 사용을 금지**합니다.

JavaScript에서 스타일을 동적으로 변경할 때도 `element.style` 대신 **클래스 토글**을 사용합니다.

```javascript
// ✗ 금지
element.style.backgroundColor = 'red';
element.style.display = 'none';

// ✓ 권장
element.classList.add('bg-danger');
element.classList.add('hidden');
```

## 고려한 대안

### 대안 1: 인라인 스타일 허용 (현상 유지)

- **장점**: 빠른 프로토타이핑, 별도 CSS 파일 불필요
- **단점**: 유지보수 어려움, 일관성 없음
- **불채택 사유**: 장기적으로 기술 부채 증가

### 대안 2: CSS-in-JS (styled-components 등)

- **장점**: 컴포넌트와 스타일 결합, 동적 스타일링 용이
- **단점**: 바닐라 JS 프로젝트에 무겁다, 학습 곡선
- **불채택 사유**: 현재 프로젝트들이 바닐라 JS 기반

### 대안 3: Tailwind CSS

- **장점**: 유틸리티 클래스로 빠른 개발
- **단점**: 클래스명이 길어짐, 빌드 필요
- **불채택 사유**: 빌드 도구 없는 프로젝트에 부적합
