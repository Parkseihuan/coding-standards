- **작성자**: 코딩 표준 위원회
- **관련 ADR**: [ADR-0002](./0002-python-src-structure.md)

## 컨텍스트

현재 Python 백엔드 프로젝트에서 Flask와 FastAPI가 혼재되어 있습니다:
- **Flask 사용**: Law-Tracking, Status-of-professors
- **FastAPI 사용**: RAG

**문제점:**
- 두 프레임워크의 문법과 패턴이 다름
- 코드 재사용 어려움
- 팀원 학습 부담 증가

## 결정

**신규 Python API 프로젝트는 FastAPI를 기본으로 사용**합니다.

FastAPI 선택 이유:
1. **타입 힌트 기반**: Pydantic으로 자동 검증
2. **자동 문서화**: Swagger UI, ReDoc 기본 제공
3. **비동기 지원**: async/await 네이티브 지원
4. **성능**: Flask 대비 높은 처리량

## 고려한 대안

### 대안 1: Flask 유지

- **장점**: 익숙함, 많은 레퍼런스
- **단점**: 타입 검증 수동, 비동기 지원 제한적
- **불채택 사유**: 현대적인 API 개발에 부적합

### 대안 2: Django REST Framework

- **장점**: 풀스택 지원, ORM 내장
- **단점**: 무겁다, 학습 곡선 높음
- **불채택 사유**: 단순 API 서버에는 과함

### 대안 3: 프레임워크별 자유 선택

- **장점**: 개발자 자율성
- **단점**: 일관성 없음, 코드 재사용 어려움
- **불채택 사유**: 표준화 목적에 반함

## 결과

- [ ] Flask만 알던 개발자 학습 필요

## 마이그레이션 가이드

Flask에서 FastAPI로 전환 시 주요 변경점:

| Flask | FastAPI |
|-------|---------|
| `@app.route('/api')` | `@app.get('/api')` |
| `request.json` | 함수 파라미터로 자동 파싱 |
| `jsonify()` | dict 반환 (자동 변환) |
| Flask-CORS | `CORSMiddleware` |
